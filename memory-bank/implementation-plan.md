# smart-filer 首版实施计划

本文件是供 AI 开发人员使用的实施计划。

当前版本不再追求完整的 `smart-filer` MVP，而是聚焦一个更小的首版目标：

- 当用户告知一个软件名字时，系统给出该软件的建议安装位置。

本文档只包含实施说明，不包含代码。

## 当前版本已确认前提

- 项目按“从零开始”处理。
- `memory-bank/progress.md` 与 `memory-bank/architecture.md` 当前为空，本版实施时忽略其内容。
- 当前不讨论数据库 schema；数据库与本地索引不是本版阻塞项。
- `文件结构.md` 中的解释性文字与 SOP 视为硬规则。
- `文件结构.md` 中的目录结构与目录名字视为可变规则数据，不应在代码中写死为不可变常量。
- 本项目依赖安装路径也必须服从 `文件结构.md` 的规则，不得随意落到不符合规划的位置。
- Python 解释器、虚拟环境、依赖安装位置应优先服从 `D:\10_Environments\Python` 的环境规划；缓存与临时文件应优先服从 `D:\00_Temp_Cache` 的规划。
- 本项目的模型调用提供方为 SiliconFlow，模型选择以 `https://cloud.siliconflow.cn/me/models` 当时可用模型为准。
- LLM 失败时，系统必须回退为保守建议，并明确要求人工处理。
- 首版不做任何自动执行操作，只需要建议逻辑正确。

## 首版目标

输入：

- 软件名称，例如 `OBS Studio`、`Altium Designer`、`7-Zip`

输出：

- 建议安装分类
- 建议安装路径
- 建议原因
- 置信度
- 是否需要人工确认
- 是否触发了回退策略
- 规则依据摘要

## 本版明确不做

- 文件路由
- 文档解析（PDF / Word / 安装包元数据提取）
- 本地数据库
- 软件清单扫描
- 按功能查软件
- 文件移动、重命名、安装程序拉起
- `watch` 监听
- 自动执行与回滚

以上能力统一延期到后续版本，不在本实施计划中展开。

## 总体执行规则

- 严格按步骤顺序实施。
- 每完成一个步骤，先补齐该步骤要求的测试，再运行测试验证。
- 若当前步骤测试未通过，不进入下一步。
- 始终保持模块化，优先拆分为多个小文件。
- `cli` 只负责命令入口与结果展示，不承载核心业务规则。
- 业务规则优先来自本地规则文档与内置硬规则，LLM 只负责辅助分类与建议。
- 一旦本地硬规则与 LLM 输出冲突，必须以前者为准。
- 项目依赖、虚拟环境、缓存路径的设计与安装步骤，必须符合 `文件结构.md` 中 `D:` 盘环境规划。
- 所有输出都必须是结构化对象，而不是自由文本。
- 本版所有结果本质上都是“建议”，不产生任何系统副作用。

## 依赖安装规则

本版不仅要求“声明依赖”，还要求“依赖安装位置正确”。

依赖安装与运行环境至少遵循以下规则：

- 项目源码位于工作目录，可继续放在 `S:\001_Workspace`。
- Python 解释器、虚拟环境、`uv` 管理的环境与依赖安装目录，应优先落在 `D:\10_Environments\Python` 下的项目专用位置。
- 依赖下载缓存、临时构建文件、工具缓存应优先落在 `D:\00_Temp_Cache` 体系。
- 不应把可迁移的运行环境直接堆放在 `S:` 盘数据体系中。
- 若具体安装方式与 `文件结构.md` 冲突，必须以 `文件结构.md` 的环境规划优先。

建议实现口径：

- 项目代码仓库与规则文档保留在工作目录。
- 项目虚拟环境命名与路径单独管理，例如为 `smart-filer` 创建独立环境目录。
- 在文档与命令示例中明确说明依赖安装路径，而不是默认使用未约束的系统全局环境。

## 建议输出模型

首版建议统一输出为结构化对象，至少包含：

```json
{
  "software_name": "OBS Studio",
  "software_category": "media_design",
  "suggested_install_path": "D:\\50_Media_Design",
  "reason": "该软件属于音视频采集与处理工具，按 D 盘软件分类规则建议安装到媒体设计目录。",
  "confidence": 0.84,
  "needs_confirmation": true,
  "fallback_used": false,
  "rule_basis": [
    "软件与运行环境优先进入 D: 盘体系",
    "媒体设计类软件 -> D:\\50_Media_Design"
  ]
}
```

约束：

- `software_name` 为用户输入或归一化后的软件名。
- `software_category` 必须来自受控枚举。
- `suggested_install_path` 必须是结构化建议，不允许为空字符串。
- `needs_confirmation` 在本版应始终为 `true`，因为系统只提供建议，不自动执行。
- `fallback_used` 用于标记是否发生 LLM 回退。
- `rule_basis` 用于说明本次建议引用了哪些规则。

## 核心规则口径

本版至少实现以下规则语义：

- 软件与运行环境优先进入 `D:` 盘体系。
- 不建议把软件安装到 `S:` 盘。
- 安装路径建议优先服从 `文件结构.md` 中的解释性文字、规则说明与 SOP。
- 目录名和目录树不是硬编码常量，应通过规则解析或配置映射得到。

首版至少支持以下软件类别到安装目录的建议映射语义：

- 开发环境或 SDK -> `D:\10_Environments`
- 工程类软件 -> `D:\30_Engineering`
- 办公效率类软件 -> `D:\40_Productivity`
- 媒体设计类软件 -> `D:\50_Media_Design`
- 系统工具类软件 -> `D:\60_System_Utilities`
- 游戏娱乐平台 -> `D:\70_Games_Entertain`

说明：

- 以上是当前规则文档中的有效语义，不代表代码中可直接写死目录名字。
- 若后续 `文件结构.md` 修改目录命名，本系统应通过重新加载规则继续工作。

## LLM 使用策略

本版 LLM 的职责收敛为：

- 根据软件名称推断软件类别
- 在类别不明确时给出低置信度分类建议

本版 LLM 的边界：

- 不直接决定最终规则
- 不绕过本地硬规则
- 不参与任何自动执行

本版调用方式约定：

- 使用 SiliconFlow 作为模型 API 提供方。
- 优先采用 SiliconFlow 官方文档支持的 OpenAI 兼容调用方式。
- 默认 API 基地址使用 `https://api.siliconflow.cn/v1`。
- 模型 ID 不在代码中写死为假定值，应通过配置指定，并以 `https://cloud.siliconflow.cn/me/models` 中当前可用模型为准。

LLM 失败回退策略：

- 若请求失败、超时、返回非结构化结果或结果无法通过校验，则进入回退逻辑。
- 回退逻辑返回保守建议：
  - 保留 `D:` 盘优先原则
  - 若无法确定类别，则给出通用保守安装位置建议或“需人工判定”的结果
  - 强制 `needs_confirmation = true`
  - 标记 `fallback_used = true`
- 回退结果必须带明确原因，不能静默失败。

测试要求：

- 验证目录结构符合分层约束。

通过标准：

- 所有基础包导入通过。
- 没有单文件承载全部职责。

### 步骤 2：建立项目元数据、依赖声明与安装规则

- 创建项目元数据文件，声明 Python 3.12 与首版依赖。

测试要求：

- 添加一个测试或校验步骤，确认依赖声明中包含本版所需依赖。

  - 项目源码位置

通过标准：

## 阶段二：规则系统

### 步骤 5：定义领域模型

- 定义软件类别枚举。
- 定义安装建议结果模型。
- 定义规则来源、规则优先级、回退状态等领域对象。
- 不使用松散字符串在系统内层层透传。

测试要求：

- 添加领域模型测试，验证枚举值、非法值处理、序列化与反序列化。
- 验证建议结果模型字段约束有效。

通过标准：

- 后续模块都依赖统一领域对象，而不是临时字典。

### 步骤 6：建立规则文档加载器

- 在基础设施层创建规则文档读取模块。
- 该模块只负责读取原始文本，不负责业务解析。
- 必须处理文档不存在、编码异常、空文件等情况。

测试要求：

- 添加加载器测试，验证可正常读取 `文件结构.md`。
- 验证中文内容读取正确。
- 验证缺失与空文件时返回明确异常。

通过标准：

- 文档读取与业务解析职责分离。

### 步骤 7：实现规则文档解析器

- 建立解析器，将 `文件结构.md` 转换为中间规则表示。
- 本版解析重点不是完整目录树，而是与“软件安装位置建议”直接相关的规则。
- 必须把解释性文字与 SOP 中的软件安装规则解析为硬规则。
- 目录名称与路径映射从文档中提取，不在业务代码中写死。

测试要求：

- 添加解析测试，验证可提取：
  - `D:` 盘优先原则
  - 软件不建议安装到 `S:` 盘
  - 软件类别到安装目录的映射
- 验证遇到未知格式时保留告警，而不是静默忽略。

通过标准：

- 规则解析结果可直接供建议服务消费。

### 步骤 8：实现安装位置硬规则服务

- 基于解析结果与内置最低限度规则，实现硬规则服务。
- 至少保证以下行为：
  - 软件建议路径只能落在 `D:` 盘软件体系
  - 若 LLM 给出 `S:` 盘路径，必须被拒绝或改写
  - 若类别存在明确映射，优先输出映射路径

测试要求：

- 添加硬规则测试，验证硬规则覆盖 LLM 决策。
- 验证非法目标路径会被纠正或拒绝。

通过标准：

- “硬规则优先”行为稳定可测。

## 阶段三：LLM 集成与回退策略

### 步骤 9：定义 LLM 请求与响应模型

- 在领域层定义 LLM 输入模型与响应模型。
- 输入至少包括：
  - 软件名称
  - 规则摘要
  - 可选的别名或上下文
- 响应至少包括：
  - 软件类别
  - 建议路径
  - 原因
  - 置信度

测试要求：

- 添加模型测试，验证字段完整与类型约束。
- 验证缺失关键字段时会被拒绝。

通过标准：

- LLM 集成从一开始就被结构化约束。

### 步骤 10：实现 SiliconFlow provider adapter

- 在基础设施层封装 SiliconFlow 调用。
- 应用层不得直接拼装 SiliconFlow HTTP 请求。
- 可使用 OpenAI Python SDK 按 SiliconFlow 官方兼容方式接入，但对外暴露的必须是项目内部的 SiliconFlow adapter 接口。
- 适配器需支持超时、错误包装、原始返回记录与模型 ID 配置。

测试要求：

- 添加适配器测试，使用 mock 验证请求组装与错误处理。
- 验证请求使用 SiliconFlow base URL 与配置中的模型 ID。
- 验证超时、空响应、非结构化响应时会返回明确项目内部错误。

通过标准：

- SiliconFlow 已被隔离在基础设施层内。

### 步骤 11：实现提示词构建器

- 创建专门模块，负责把软件名与规则摘要拼成提示词输入。
- 提示词构建器与 provider adapter 分离。
- 明确要求模型输出结构化结果，不允许自由散文式回答。

测试要求：

- 添加提示词构建测试，验证输入信息完整、顺序稳定。
- 验证规则摘要会被显式注入提示词。

通过标准：

- 提示词逻辑可独立迭代。

### 步骤 12：实现 LLM 响应解析与回退服务

- 接收 provider adapter 返回内容并解析为统一建议模型。
- 若返回无效、类别冲突、路径非法或低置信度，则进入回退策略。
- 回退服务需返回保守建议并要求人工处理。

测试要求：

- 添加解析与回退测试，覆盖：
  - 合法结构化响应
  - 非法 JSON
  - 字段缺失
  - 低置信度
  - 路径落在 `S:` 盘
  - LLM 完全失败

通过标准：

- 任意异常 LLM 结果都不会直接成为最终建议。

## 阶段六：验收与文档整理

### 步骤 17：补齐集成测试

- 建立集成测试，覆盖从 CLI 到应用层到规则系统再到 LLM 回退逻辑的主路径。
- 集成测试优先使用真实模块拼装，只在 SiliconFlow 调用处使用替身。

测试要求：

- 覆盖以下关键场景：
  - 正常分类并返回路径
  - LLM 返回非法结构
  - 规则文档缺失
  - 规则文档修改后重新加载生效

通过标准：

- 主路径在集成层面可重复通过。

### 步骤 18：整理开发文档与验收清单

- 更新 README 或相关文档，说明：
  - 当前版本只支持“软件安装位置建议”
  - 输入输出格式
  - 规则来源
  - 依赖安装路径规则
  - SiliconFlow 配置项与模型来源
  - LLM 回退策略
  - 明确未实现能力
- 整理本版验收清单。

测试要求：

- 人工校验文档与当前实现一致。
- 至少执行一次“按文档说明运行命令并得到结果”的验证。

通过标准：

- 新的开发人员可仅依靠文档理解本版边界与实现路径。

## 本版完成定义

当且仅当以下条件全部满足时，本计划视为完成：

- 用户可输入一个软件名字。
- 系统可返回结构化的安装位置建议。
- 建议结果始终符合 `D:` 盘优先与“不安装到 `S:` 盘”的硬规则。
- `文件结构.md` 的解释性文字与 SOP 已被纳入规则来源。
- LLM 失败时系统会返回保守建议，并明确要求人工处理。
- 全流程不包含任何自动执行操作。
- 关键路径具备单元测试与集成测试支撑。
- 文档、实现、测试三者一致。
