# smart-filer 设计文档 / PRD

## 1. 产品概述

`smart-filer` 是一个面向 Windows 本地环境的文件智能路由工具。它的目标是读取本地文件、安装程序与当前电脑文件结构，结合 [`文件结构.md`](./文件结构.md) 中定义的规则，并调用 LLM API，为文件或安装程序生成“应放到哪里、应该叫什么、是否需要安装、建议安装到哪里”的智能决策。

该工具的核心定位不是“完全替代人工”，而是作为一个本地智能分拣助手，帮助用户降低文件整理、软件安装与软件查找的决策成本。

---

## 2. 产品目标

### 2.1 目标

1. 对新进入系统的文件进行自动识别、分类、命名与路由。
2. 对安装程序进行识别，给出合适的安装位置或安装建议。
3. 实时维护当前文件结构与软件清单，支持“要实现某个功能时，先判断本机是否已有对应软件”。
4. 将所有判断结果尽量与 `文件结构.md` 中的规则保持一致。

### 2.2 非目标

1. 不在首版中实现复杂的文件内容深度编辑。
2. 不在首版中实现静默安装所有软件。
3. 不在首版中替代系统级包管理器，仅提供建议或辅助执行。

---

## 3. 目标用户

- 重视本地文件结构管理的个人用户
- 经常下载 PDF、Word、安装包、压缩包、工程资料的知识工作者
- 希望建立“数据归 S 盘、执行归 D 盘”体系的 Windows 用户

---

## 4. 使用场景

### 场景 1：新文件进入系统

用户下载或接收一个 PDF、Word、图片、压缩包、脚本或其他文件后，`smart-filer` 识别文件类型、文件名、来源信息与可能用途，决定其目标目录，并按规则生成标准命名。

### 场景 2：安装程序处理

用户下载一个 `.exe`、`.msi` 或压缩安装包后，`smart-filer` 判断其属于工程类、办公类、媒体类、系统工具类或游戏类软件，并根据 `D:` 盘规则给出建议安装目录。

### 场景 3：功能需求驱动的软件检查

当用户提出“我要录屏”“我要编辑 PDF”“我要抓串口数据”“我要写 Python 脚本”等需求时，系统先检查电脑上是否已有可用软件；若没有，则调用 LLM API 给出推荐软件及建议安装位置。

### 场景 4：安装包归档

软件安装后，系统判断安装包是否值得长期保留。若属于稀缺版本、大体积或版本锁定软件，则建议归档到 `S:\500_Database\550_Installers_Archive`；否则建议删除。

---

## 5. 核心原则

`smart-filer` 的所有路由判断默认遵循以下原则：

- 数据文件优先进入 `S:` 盘体系。
- 软件与运行环境优先进入 `D:` 盘体系。
- 所有外来未处理文件优先进入 `S:\000_Inbox`。
- 高频处理中内容优先进入 `S:\001_Workspace`。
- 可跨项目复用的内容优先进入 `S:\500_Database`。
- 文件命名应遵循 `文件结构.md` 中定义的命名规则。

---

## 6. 功能需求

### 6.1 文件智能识别

系统应支持识别以下信息：

- 文件扩展名，如 `pdf`、`docx`、`xlsx`、`exe`、`msi`、`zip`
- 文件名中的主题、时间、版本、作者、项目名等线索
- 文件基础元数据，如大小、创建时间、修改时间
- 可选的文件内容摘要能力
  - PDF：提取标题、第一页文本、作者等
  - Word：提取标题、正文前若干段
  - 安装程序：识别产品名、版本、厂商、平台

### 6.2 智能路由决策

系统应根据本地规则和 LLM 判断，输出：

- 推荐目标路径
- 推荐新文件名
- 决策原因
- 置信度
- 是否建议人工确认

### 6.3 安装程序处理

系统应支持：

- 识别安装程序类别
- 推荐安装位置
- 推荐是否保留安装包
- 对绿色软件与正式安装软件做区分

建议映射如下：

- 工程类软件 -> `D:\30_Engineering`
- 办公效率类软件 -> `D:\40_Productivity`
- 媒体设计类软件 -> `D:\50_Media_Design`
- 系统工具类软件 -> `D:\60_System_Utilities`
- 游戏娱乐平台 -> `D:\70_Games_Entertain`
- 开发环境或 SDK -> `D:\10_Environments`

### 6.4 文件命名规范化

系统应支持按文件类型自动套用命名模板：

- 通用文档：`标题_作者(可选)_V版本号_YYYYMMDD`
- 时间流文件：`YYYYMMDD_事项名_补充说明`
- 安装包归档：`软件名_版本_平台_备注`

### 6.5 本地文件结构实时记录

系统应维护一份本地索引，用于记录：

- 已扫描目录结构
- 常用文件目录
- 已归档安装包
- 已安装软件清单
- 文件处理历史

### 6.6 软件存在性检查

当用户提出一个功能目标时，系统应先检查本机是否已有相关软件。检查来源可包括：

- 已知安装目录扫描
- Windows 开始菜单快捷方式
- 注册表卸载信息
- 用户自定义软件清单

输出结果应包括：

- 是否已有对应软件
- 已有软件的位置
- 是否满足当前需求
- 若不满足，则推荐候选软件及安装位置

### 6.7 操作模式

首版建议支持两种模式：

- 建议模式：只输出建议，不自动移动、不自动安装
- 执行模式：在用户确认后移动文件、重命名文件，或拉起安装程序

---

## 7. 核心流程

### 7.1 文件路由流程

1. 监听或接收用户指定文件
2. 读取文件元数据与基础内容
3. 加载 `文件结构.md` 中的目录规则与命名规则
4. 查询本地索引，判断是否已有类似文件或项目上下文
5. 调用 LLM API 生成分类、命名与路径建议
6. 进行规则校验
7. 输出建议或执行移动/重命名
8. 写入处理日志与索引

### 7.2 安装程序流程

1. 识别安装程序类型与软件类别
2. 检查本机是否已安装同类软件或同名软件
3. 结合 `D:` 盘规则生成建议安装位置
4. 判断安装包是否需要归档
5. 输出建议，必要时等待用户确认后执行

### 7.3 功能需求查询流程

1. 用户输入目标，如“我要剪视频”
2. 系统将目标映射为软件能力标签
3. 检查本地软件索引
4. 若已存在，则返回软件名称、路径与建议使用项
5. 若不存在，则调用 LLM API 推荐软件并给出安装目录建议

## 8. 系统模块设计

### 8.1 规则引擎

职责：

- 解析 `文件结构.md`
- 固化高优先级硬规则
- 对 LLM 输出进行约束和校验

示例硬规则：

- 新外来文件默认先到 `S:\000_Inbox`
- 活跃项目文件优先 `S:\001_Workspace`
- 软件不直接建议安装到 `S:` 盘

### 8.2 文件分析模块

职责：

- 读取文件元数据
- 调用对应解析器提取文本与标题
- 对安装包做产品信息识别

### 8.3 LLM 决策模块

职责：

- 将规则、文件信息、上下文拼装为提示词
- 输出分类、命名、路径、解释与置信度
- 在结果不确定时触发人工确认

### 8.4 本地索引模块

职责：

- 保存目录结构快照
- 保存文件处理历史
- 保存已安装软件清单
- 保存软件能力标签映射

建议使用：

- SQLite 或 JSON 作为首版存储

### 8.5 执行模块

职责：

- 文件移动
- 文件重命名
- 安装程序调用
- 归档记录写入

### 8.6 交互层

建议首版支持：

- CLI 命令行模式
- 后续可扩展 Tray 常驻服务或简易桌面 GUI

---

## 9. LLM 输入输出设计

### 9.1 输入

- 文件名
- 文件路径
- 文件类型
- 文件摘要
- 本地目录规则摘要
- 当前目录结构上下文
- 已安装软件清单
- 用户当前任务或意图

### 9.2 输出

建议统一为结构化 JSON：

```json
{
  "category": "document | installer | media | utility",
  "suggested_path": "S:\\300_Life\\340_Logistics",
  "suggested_name": "20260415_家电说明书_空气炸锅.pdf",
  "reason": "文件内容属于生活后勤类资料，适合长期归档到 Logistics",
  "confidence": 0.92,
  "needs_confirmation": false,
  "software_recommendation": null
}
```

## 10. MVP 范围

首版建议优先完成以下能力：

1. 读取 `文件结构.md` 并转成可用规则
2. 支持对 `pdf`、`docx`、`exe`、`msi`、`zip` 做识别
3. 输出推荐路径与推荐命名
4. 支持“建议模式”
5. 记录处理日志
6. 支持按功能查询本机是否已有相关软件
7. 若没有，则给出软件推荐和安装路径建议

---

## 11. 非功能需求

- 默认本地优先，避免上传完整私人文件内容
- 对敏感文件支持只上传摘要而非全文
- 所有自动执行操作都应支持预览与回滚
- 处理日志应可追溯
- 规则变更后应能重新生效

---

## 12. 风险与注意事项

- LLM 可能误判文件语义，因此高风险操作必须保留人工确认
- 安装程序类别识别可能不准确，需要结合规则和关键字双重判断
- 文件移动可能影响其他软件引用路径，因此应优先在建议模式验证
- 本地软件扫描来源较多，首版需容忍信息不完整

---

## 13. 后续扩展方向

- 文件夹级批量整理
- 下载目录自动监听
- 与 Everything、Windows Search 或本地向量索引集成
- 自动识别重复文件与历史版本
- 基于用户反馈持续优化提示词与规则
- GUI 可视化确认面板

---

## 14. 一句话定义

`smart-filer` 是一个基于本地目录规则与 LLM 决策的 Windows 文件智能路由助手，用来帮用户决定文件该放哪、该叫什么、软件是否已有、没有时该装什么以及装到哪里。
